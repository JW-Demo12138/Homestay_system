# 基于SpringBoot+Vue的乡村旅游民宿预订平台开发文档
## 一、文档概述
### （一）文档目的
本文档基于开题报告核心需求，明确平台开发的技术架构、功能实现方案、开发规范及测试要求，为开发团队提供标准化的开发指引，确保平台按计划实现“自主可控、适配乡村、融合特色”的核心目标。

### （二）文档范围
覆盖平台从技术选型、架构设计、模块开发到测试部署的全流程，适用于后端开发、前端开发、测试及运维人员，作为开发实施与质量管控的核心依据。

## 二、项目核心信息
### （一）项目目标
1. 技术目标：构建前后端分离、轻量化、易部署维护的民宿预订系统，适配乡村非技术背景用户使用场景。
2. 功能目标：实现用户管理、民宿管理、预订支付、地理信息可视化、标签筛选、多角色协同等核心功能。
3. 应用目标：帮助乡村民宿经营者建立自主数字化运营体系，助力乡村振兴。

### （二）核心技术栈
1. 后端技术：SpringBoot（核心框架）、MyBatis（持久层框架）、MySQL（数据库）、Leaflet（WebGIS技术）、JDK 1.8+。
2. 前端技术：Vue.js（核心框架）、Vue CLI（脚手架）、Element UI（UI组件库）、Axios（网络请求工具）、Mars3D（GIS可视化插件）。
3. 开发与测试工具：IntelliJ IDEA、Visual Studio Code、Postman（接口测试）、MySQL Workbench（数据库管理）、Git（版本控制）。

## 三、技术架构设计
### （一）整体架构
采用前后端分离架构，分为前端层、接口层、服务层、数据层四层结构：
1. 前端层：负责页面渲染、用户交互及GIS可视化展示，通过Axios与后端接口通信。
2. 接口层：基于SpringBoot实现RESTful API，处理跨域、请求拦截、响应封装等通用逻辑。
3. 服务层：实现核心业务逻辑，包括用户服务、民宿服务、订单服务、权限服务等模块。
4. 数据层：通过MyBatis操作MySQL数据库，整合数据库优化策略，确保数据存取高效稳定。

### （二）数据库设计
#### 1. 核心数据表
| 表名 | 核心字段 | 设计说明 |
|------|----------|----------|
| user | id、username、password、role（角色：房东/游客/管理员）、phone、status | 存储用户基础信息，区分角色权限 |
| homestay | id、name、address、latitude（纬度）、longitude（经度）、price、room_type、facility、status、owner_id | 存储民宿基础信息，关联房东ID |
| homestay_feature | id、homestay_id、content（特色描述）、image_url1-3、tag | 存储民宿特色内容，关联民宿ID |
| order | id、user_id、homestay_id、check_in_date、check_out_date、price、status、pay_time | 存储订单信息，关联用户与民宿 |
| surrounding_resource | id、name、type（景点/采摘区等）、latitude、longitude、homestay_id | 存储民宿周边资源，支持GIS可视化 |
| complaint | id、user_id、homestay_id、content、status、handle_time | 存储游客投诉反馈信息 |

#### 2. 优化策略
- 采用索引优化：在homestay表的address、latitude、longitude字段建立索引，提升GIS空间检索效率。
- 数据分割：对订单表按时间分区存储，降低查询压力。
- 连接池配置：基于MyBatis连接池优化方案，配置异步连接创建与回收机制，提升持久层性能。

### （三）核心技术实现方案
#### 1. GIS可视化实现
- 集成Leaflet开源WebGIS技术，前端生成交互式电子地图，加载民宿及周边资源的地理坐标数据。
- 实现地图缩放、平移功能，支持民宿位置高亮显示，关联周边资源标签弹窗。

#### 2. 多角色权限控制
- 基于Spring Security实现权限管控，通过角色分配不同接口访问权限。
- 房东仅可操作自身民宿及订单数据，管理员拥有全量审核与监管权限，游客仅可访问预订相关功能。

#### 3. 特色内容上传与审核
- 房东上传特色内容时，限制图片数量（最多3张），支持预设标签选择。
- 管理员审核流程：待审核→审核通过（展示）/审核驳回（反馈原因）。

## 四、功能模块开发
### （一）用户管理模块
#### 1. 开发内容
- 注册功能：手机号验证、密码强度校验（至少8位，包含字母+数字），角色选择（房东/游客）。
- 登录功能：手机号+密码登录，生成JWT令牌，有效期2小时。
- 个人信息管理：支持昵称、头像、联系方式修改，房东额外支持经营资质补充。

#### 2. 接口设计
- POST /api/user/register：用户注册
- POST /api/user/login：用户登录
- GET /api/user/info：获取个人信息
- PUT /api/user/info：修改个人信息

### （二）民宿管理模块
#### 1. 开发内容
- 房源发布：房东填写民宿基础信息，上传图片（支持批量上传，单张不超过5MB），选择配套设施。
- 房源编辑与下架：支持民宿信息修改，下架后不可被搜索，保留历史数据。
- 特色内容上传：房东可选填特色描述，上传最多3张图片，选择预设标签（如“客家文化”“田园风光”）。
- 管理员审核：审核民宿基础信息及特色内容，审核结果实时反馈给房东。

#### 2. 接口设计
- POST /api/homestay：发布民宿
- PUT /api/homestay/{id}：编辑民宿
- DELETE /api/homestay/{id}：下架民宿
- POST /api/homestay/feature：上传特色内容
- PUT /api/homestay/audit/{id}：管理员审核民宿

### （三）搜索与筛选模块
#### 1. 开发内容
- 地理信息可视化查询：地图展示民宿分布，标注周边资源（景点、采摘区等），点击民宿显示详情弹窗。
- 标签筛选：提供乡村特色主题标签列表，用户选择后，系统通过GIS空间检索筛选符合条件的民宿，地图高亮显示。

#### 2. 接口设计
- GET /api/homestay/map：获取民宿地理坐标及周边资源数据
- GET /api/homestay/filter：标签筛选民宿（参数：tag、latitude、longitude）

### （四）预订与订单模块
#### 1. 开发内容
- 在线预订：游客选择入住/退房日期，系统校验房源可用性，生成订单。
- 在线支付：集成第三方支付接口（如支付宝沙箱环境），支付成功后更新订单状态。
- 订单管理：游客可查看订单状态、取消预订（未支付订单可直接取消，已支付订单按取消政策处理）；房东可查看订单、确认订单。
- 评价与投诉：游客完成订单后可评价民宿，支持文字评价；若有纠纷可提交投诉反馈。

#### 2. 接口设计
- POST /api/order：创建订单
- POST /api/order/pay/{id}：订单支付
- GET /api/order/user：用户查询个人订单
- GET /api/order/owner：房东查询订单
- POST /api/order/comment：提交订单评价
- POST /api/order/complaint：提交投诉反馈

### （五）管理员模块
#### 1. 开发内容
- 用户管理：查看平台所有用户，禁用违规用户账号。
- 资源审核：审核民宿信息及房东上传的特色内容，驳回不合规内容并反馈原因。
- 投诉处理：查看游客投诉，处理后更新投诉状态，反馈处理结果。

#### 2. 接口设计
- GET /api/admin/user：查询所有用户
- PUT /api/admin/user/{id}/status：修改用户状态（启用/禁用）
- GET /api/admin/homestay/pending：查询待审核民宿
- PUT /api/admin/feature/audit/{id}：审核特色内容
- PUT /api/admin/complaint/{id}：处理投诉

## 五、开发规范
### （一）代码规范
1. 后端：遵循Alibaba Java Coding Guidelines，类名采用PascalCase，方法名、变量名采用camelCase，注释覆盖率≥60%。
2. 前端：组件名采用PascalCase，Props命名采用camelCase，CSS使用Scoped隔离样式，避免全局污染。
3. 数据库：表名、字段名采用snake_case，主键统一命名为id，外键命名格式为“关联表名_id”。

### （二）版本控制规范
- 采用Git分支管理：master（主分支）、develop（开发分支）、feature/xxx（功能分支）、hotfix/xxx（紧急修复分支）。
- 提交信息格式：[模块名] 操作描述（如：[用户模块] 新增注册功能）。

### （三）接口规范
- 统一响应格式：{"code": 200,"message": "成功","data": {}}，错误码按模块划分（如用户模块错误码1001-1999）。
- 接口请求方式：查询用GET，新增用POST，修改用PUT，删除用DELETE。
- 分页参数：统一使用page（页码）、size（每页条数），默认page=1，size=10。

## 六、测试计划
### （一）测试类型
1. 单元测试：对核心业务逻辑（如订单生成、权限校验）编写单元测试用例，覆盖率≥70%。
2. 接口测试：使用Postman测试所有API，验证参数校验、响应格式、权限控制是否符合要求。
3. 功能测试：模拟用户场景，测试核心流程（注册-登录-搜索-预订-支付-评价）的完整性。
4. 性能测试：测试并发场景下（如100人同时查询民宿）系统响应时间，要求≤3秒。
5. 兼容性测试：兼容Chrome、Firefox、Edge等主流浏览器，支持PC端及移动端自适应。

### （二）测试重点
- GIS可视化功能：地图加载速度、筛选准确性、坐标显示正确性。
- 支付流程：支付状态同步、订单状态更新及时性。
- 权限控制：不同角色接口访问权限校验，防止越权操作。

## 七、部署与维护
### （一）部署方案
1. 环境要求：JDK 1.8+、MySQL 8.0+、Nginx（前端部署）、小型云服务器（2核4G以上）。
2. 部署流程：
- 后端：通过Maven打包为Jar包，使用SpringBoot内置Tomcat部署，配置application.yml文件（数据库连接、端口等）。
- 前端：执行npm run build打包，将dist目录部署到Nginx，配置反向代理指向后端接口。
- 数据库：导入SQL脚本，执行索引创建、分区配置等优化操作。

### （二）维护方案
1. 日常维护：定期备份数据库（每日自动备份），查看系统日志（使用Logback记录日志）。
2. 故障处理：模块化架构支持单独模块重启，通过日志定位故障点，快速修复。
3. 功能迭代：预留接口扩展点，支持后期新增功能（如农产品推荐），无需重构核心架构。
